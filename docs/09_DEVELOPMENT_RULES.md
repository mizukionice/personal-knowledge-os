# 09_DEVELOPMENT_RULES — 開発ルール

## 必ず守る
1. Task単位で実装しcommitする。commitメッセージ: `M{x}-{nn}: 内容`。完了したら08_TASKS.mdのチェックを更新
2. テストを書いてから実装する。最低限: packages/kpsの各ステージ（外部APIはモック）、apiのハンドラ、共有zodスキーマ
3. `pnpm lint && pnpm typecheck && pnpm test` が通らない状態でcommitしない
4. README.md（セットアップ・運用手順）を変更のたびに更新する
5. ADR（02_ADR.md）を勝手に変更しない。ライブラリ追加はまず提案する
6. KPS（04_KPS.md）のインターフェース境界を壊さない。実装の追加は可、境界の変更は提案
7. secretsをコード・ログに出さない。.env.exampleを最新に保つ
8. DBスキーマ変更は必ずsupabase/migrations/にSQLファイルを追加（直接変更しない）
9. RLS・user_id分離を全テーブル・全クエリで維持する
10. 動くものを優先する。過度な抽象化・先回り実装をしない（YAGNI）

## コード規約
- TypeScript strict。`any`禁止（やむを得ない場合はコメントで理由）
- 外部I/O（VLM/DB/R2/GitHub）は必ずインターフェース経由にし、テストでモック可能にする
- エラーは握りつぶさない。バッチはjobs.error/pages.errorに記録して失敗を可視化
- UIテキストは日本語

## 進め方
- 各マイルストーン開始時に、対象タスクの実装方針を3〜5行で要約してから着手する
- 詰まったら仕様書参照順（CLAUDE.md）に従う。それでも不明ならユーザーに質問する
- 大きな変更のあとは動作確認手順を提示する（ユーザーが手元で確認できるように）
